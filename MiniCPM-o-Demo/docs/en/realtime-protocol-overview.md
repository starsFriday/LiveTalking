# MiniCPM-o Realtime API Protocol

This document records the current Realtime API implementation. The public
documentation source lives under `docs-app/content/docs/en/realtime-api/`, and the Gateway image
builds it into `/docs`.

Public clients connect only to the Gateway. The actual chain is:

```text
Client -> Gateway -> Python Worker -> Backend
```

The Gateway handles queueing, worker assignment, session recording, and WebSocket forwarding. The
Worker exposes internal runtime WebSockets and forwards `session.init` / `input.append` /
`session.close` to the backend, either C++ `llama-omni-server` or the PyTorch backend.

## Endpoint

```text
wss://host/v1/realtime?mode={chat|video|audio}
```

| Parameter | Required | Description |
|-----------|----------|-------------|
| `mode` | No | Defaults to `video`; allowed values are `chat`, `video`, and `audio` |

`session_id` is generated by the server and returned in events. Clients do not pass it in the URL
and should not rely on a specific format.

## Modes

| Mode | Input | Output | Notes |
|------|-------|--------|-------|
| `chat` | One turn of `messages` | Text deltas, optional audio, `response.done` | Supports streaming and non-streaming |
| `video` | Continuous audio, optionally with video frames | `listen`, text deltas, audio deltas | 300 second session limit |
| `audio` | Continuous audio | `listen`, text deltas, audio deltas | 600 second session limit |

`chat` maps to backend runtime mode `turn_based`; both `video` and `audio` map to `full_duplex`.

## Event Model

Client events:

- `session.init`: initializes the session. The message must contain an object-valued `payload`.
- `input.append`: submits model input. The message must contain an object-valued `input`.
- `session.close`: closes the session.

Server events:

- `session.queued` / `session.queue_update` / `session.queue_done`: queue and worker assignment state.
- `session.created`: initialization completed.
- `response.output.delta`: unified output event, with `kind` set to `listen`, `text`, or `audio`.
- `response.done`: chat mode only; marks one turn response as completed.
- `session.closed`: session closed.
- `error`: error event.

`session.created.mode` is the backend runtime mode, not the public API mode from the URL.

## Lifecycle

```text
Client connects
  <- session.queued / session.queue_update   optional
  <- session.queue_done
  -> session.init
  <- session.created
  -> input.append
  <- response.output.delta / response.done
  -> session.close
  <- session.closed
```

Clients should wait for `session.queue_done` before sending `session.init`. If no queueing is
needed, the server sends `session.queue_done` immediately.

## Output Kinds

One `response.output.delta` represents exactly one output branch:

| `kind` | Fields | Description |
|--------|--------|-------------|
| `listen` | `metrics` | The model continues listening for user input |
| `text` | `text` | Text delta |
| `audio` | `audio` | 24 kHz mono float32 PCM, base64-encoded |

Text and audio are independent deltas and are not guaranteed to align one-to-one. Clients should
update captions and audio playback separately in receive order.
